Agent JSON File — In Detail

Every Kiro agent is a single JSON file. Here's what each key does:

{ "name": "trello", "description": "Creates Trello cards", "prompt": "You are a PM agent...", "tools": ["shell", "read", "write"], "allowedTools": ["shell", "read"], "resources": [], "hooks": { "postToolUse": [{ ... }] }, "welcomeMessage": "👋 Hello!" }
KeyWhat it does
nameAgent identifier — invoke with @name
descriptionShort summary shown when listing agents
promptThe full system prompt — defines personality, rules, and behaviour
toolsTools the agent can use: shell, read, write, grep, glob, aws, etc.
allowedToolsSubset of tools auto-approved without asking user permission
resourcesFiles or URLs loaded as context when agent starts
hooksShell commands that run automatically at trigger points
welcomeMessageFirst message shown when agent is activated

Key insight: The prompt is where all the magic lives — it's the detailed instruction set that makes the agent behave like a specialist.

2

What are Kiro Hooks?

Shell commands that run automatically at specific trigger points during an AI conversation.

agentSpawn

When agent starts. Load context — git status, project info, environment health.

userPromptSubmit

When user sends a message. Inject dynamic context or log prompts.

preToolUse

Before a tool runs. Can block execution (exit code 2). Security gate.

postToolUse

After a tool runs. Auto-format, log, capture output, trigger side effects.

stop

After assistant responds. Run tests, auto-commit, generate docs.

Key Difference

Hooks are deterministic shell scripts. Agents are LLM-powered. Hooks can't reason — they execute.

Agent 1: @trello

A senior product manager that creates well-structured Trello cards with BDD scenarios.

What it does

  • Asks 5 questions — one at a time
  • Reads your repo to understand tech stack
  • Generates user stories with Given/When/Then
  • Auto-splits stories > 5 SP
  • Creates cards via Trello API with labels
  • Waits for approval before creating

Card Structure

  • Title: [BE]/[FE] prefix, max 80 chars
  • User Story: As a / I want / So that
  • Business Scenarios (BDD)
  • Tasks checklist
  • Acceptance Criteria
  • Implementation Plan (if > 3 SP)

Trello Agent — Key Design Decisions

🎯 One Question at a Time

Rule #1 in the prompt. Prevents overwhelming the user. Creates a natural conversation flow instead of a form dump.

✂️ Auto-Split Large Stories

Stories > 5 SP are automatically broken into smaller deliverable units. Each sub-story is independently testable.

🔒 Approval Gate

Never creates a card without explicit user approval. Shows the full card first, waits for confirmation.

📖 Context-Aware

Reads the project repo to understand the tech stack before crafting the story. Stories reference actual frameworks and patterns.

🏷️ Auto-Labelling

Assigns [BE]/[FE] labels automatically. Creates labels on the board if they don't exist yet.

🔧 Env Var Credentials

API keys stored in shell profile as env vars. Never hardcoded in the agent config. Setup script handles this.

Trello Agent — Learnings & Iterations

The agent didn't work perfectly on day one. Here's how we iterated:

Problem: Giant user stories

Initially the agent created massive stories with 10+ tasks, touching both BE and FE. Too big to deliver in one sprint.

Fix: Auto-split rule

Added "stories > 5 SP must be split into smaller deliverable units." Now it automatically breaks them down and shows all sub-stories for approval.

Problem: Question dump

Agent asked all 5 questions in a single message. Felt like filling a form, not having a conversation.

Fix: Rule #1 — one question at a time

Made it the highest priority rule in the prompt. Added the "📋 Question X of Y" format to enforce structure.

Problem: Tasks before scenarios

Agent jumped straight to implementation tasks. Business scenarios (Given/When/Then) were missing or came last.

Fix: Strict section ordering

Enforced: User Story → Business Scenarios → Tasks → Acceptance Criteria. BDD scenarios must come before any technical details.

Problem: Created cards without asking

Agent sometimes created the Trello card immediately without showing it first. No chance to review or edit.

Fix: Approval gate

Added explicit rule: "Never create a card without approval. Show the full card in markdown first. WAIT for confirmation."

Takeaway: Agent prompts need to be iterated. Start simple, observe what goes wrong, add specific rules to fix each behaviour. The more precise the constraint, the better the output.

Agent 2: @test-reporter

A QA reporting agent that generates stakeholder-friendly test reports with charts and PDF output.

6-Step Pipeline

  • Step 1: Run tests & collect all data
  • Step 2: Classify defects in plain English
  • Step 3: Generate self-contained HTML report
  • Step 4: Convert to PDF (puppeteer-cli)
  • Step 5: Update history.json for trends
  • Step 6: Confirm output to terminal

Report Sections

  • Executive Summary (go/no-go in 90 sec)
  • Test Execution Summary + Charts
  • Defect Summary + Severity Table
  • Test Coverage Analysis
  • Risk Assessment
  • Environment & Configuration
  • Recommendations & Next Steps
6

Test Reporter — Verdict System

Every report ends with a clear release recommendation:

✅ Safe to Release

Pass rate ≥ 95%
AND 0 critical defects

GREEN

⚠️ Release with Caution

Pass rate 85–94%
OR 1–2 critical defects

AMBER

🛑 Do Not Release

Pass rate < 85%
OR 3+ critical defects

RED

Writing Rule: Never say "92% pass rate" alone — say "which means 9 in every 10 features are working as expected." Every number must have meaning for a business reader.

8

Test Reporter — Learnings & Iterations

Building this agent taught us about the boundary between hooks and agents:

Problem: Hook tried to generate the report

First version — the hook itself generated a markdown report using bash. Output was basic tables with no analysis, no charts, no defect classification.

Fix: Hook collects data, agent generates report

Separated concerns. Hook captures test output + metadata. Agent (with LLM intelligence) does the analysis, translation, and HTML generation.

Problem: Hook on wrong agent

Put the hook on the test-reporter agent itself. But the agent's prompt already tells it to generate reports — making the hook redundant.

Fix: Understood when hooks add value

Hooks add value when they do something the agent's prompt doesn't cover — like auto-triggering from a different agent, or blocking/logging silently.

Problem: Skipped report on test failures

Agent sometimes asked "tests failed, should I still generate a report?" instead of just doing it.

Fix: "ALWAYS generate" rule

Added explicit instruction: "Report generation must NEVER be skipped or made conditional on test results. Even if all tests fail, produce the report."

Problem: Technical jargon in reports

Early reports included raw stack traces, class names, and HTTP status codes. Useless for business stakeholders.

Fix: Translation table + writing rules

Added explicit error-to-English mappings and 5 strict writing rules. "NullPointerException" → "Part of the app crashed unexpectedly."

Takeaway: If it needs LLM reasoning (classify, summarize, translate) → it's an agent's job. If it's deterministic (format, block, log) → it's a hook's job. Don't mix them.

10

Key Learnings 💡

1. Iterate on prompts

Agent prompts don't work perfectly on day one. Start simple, observe what goes wrong, add specific rules to fix each behaviour.

2. One agent, one job

Combining too many responsibilities in one agent dilutes quality. Better to have focused agents and switch between them.

3. Explicit rules beat vague instructions

"Ask one question at a time" works. "Be conversational" doesn't. The more precise the constraint, the better the output.

4. Always add an approval gate

For agents that take real-world actions (API calls, card creation, deployments) — always show the output first and wait for user confirmation.

5. Context-awareness matters

Reading the project repo before generating stories or reports makes the output significantly more relevant and accurate.

6. Translate for your audience

Technical output is useless for stakeholders. Explicit translation rules ("NullPointerException" → "app crashed") make reports actually useful.

Decision Framework

When to use an agent vs a hook:

NeedUseWhy
Generate a test report with analysisAgentNeeds LLM to classify defects, write summaries
Auto-format code after writeHookDeterministic — just run prettier
Block writes to .env filesHookSimple path check, exit code 2
Create Trello storiesAgentNeeds conversation, reasoning, API calls
Scan for hardcoded secretsHookRegex scan, no reasoning needed
Load git status on startHookagentSpawn — inject context automatically
Debug failing testsAgentNeeds LLM to analyze errors and suggest fixes
Log every tool call for auditHookAppend to file — no intelligence needed

Rule of thumb: If it needs reasoning → Agent. If it's a script → Hook.
