What is a Vector Database?

A database that stores and searches data by meaning, not exact keywords

📦 What it stores

High-dimensional vectors (arrays of numbers) that represent the semantic meaning of text, images, or any data.

🔍 How it searches

Finds the most similar vectors using distance metrics (cosine, euclidean) — not exact string matching.

🧠 Why it matters

"cricket rules" matches "batting regulations" because they're semantically close — impossible with SQL LIKE.

Traditional DB: SELECT * WHERE text LIKE '%cricket%' → exact match only

Vector DB: Find nearest neighbors to embed("cricket rules") → meaning-based match

Search typeKeyword / exact matchSemantic similarity
Understands synonymsNoYes

ExamplePostgreSQL, MySQLpgvector, Pinecone, Weaviate

💡 We use pgvector — a PostgreSQL extension. No new infrastructure needed, just add vector columns to your existing DB.

How Embeddings Work

Text → Numbers that capture meaning

"King" → [0.21, 0.85, 0.12, 0.67, ..., 0.43] // 768 dimensions "Queen" → [0.19, 0.82, 0.14, 0.71, ..., 0.41] // very similar! "Apple" → [0.91, 0.03, 0.78, 0.11, ..., 0.88] // completely different

🔢 What's a dimension?

Each number captures one aspect of meaning: royalty, gender, power, food, etc.

📐 How to compare?

Cosine similarity — measures the angle between two vectors. Closer = more similar.

🤖 Who creates them?

Embedding models: nomic-embed-text (Ollama), OpenAI text-embedding-3-small

6

How are vectors compared? Cosine Similarity

Measures the angle between two vectors — not the distance

Smaller angle = more similar meaning

📐 The Formula

cos(θ) = (A · B) / (|A| × |B|) Result range: -1 to 1 1.0 = identical meaning 0.0 = unrelated -1.0 = opposite meaning

🔢 Example Calculation

A = "King" → [0.9, 0.8] B = "Queen" → [0.85, 0.9] A · B = (0.9×0.85)+(0.8×0.9) = 1.485 |A| = √(0.81+0.64) = 1.204 |B| = √(0.72+0.81) = 1.237 cos(θ) = 1.485/(1.204×1.237) = 0.997 ✓ Very similar!

Phase 1: RAG Flow — End to End

📥 Ingestion (one-time)

POST /api/ingest (file: rules.pdf) ↓ Extract text (PDFBox) ↓ Split into chunks (500 chars) ↓ Embed each chunk (Ollama) ↓ Store in pgvector

🔍 Query (every request)

POST /api/chat "What are the rules?" ↓ Embed question (Ollama) ↓ Search pgvector (top 5 similar) ↓ Pass chunks as context to LLM ↓ LLM answers from YOUR docs

💡 The LLM never sees the full document — only the most relevant chunks. This is how RAG scales to thousands of pages.

A Different Retrieval Strategy

Vectorless RAG

When Vector Search isn't the right tool — and what to use instead

Rishabh Gupta · May 2026

BM25 Keyword Search Real Examples
18

RAG has Multiple Retrieval Strategies

The "Retrieve" step in RAG is not one-size-fits-all. Vector search is just one option.

🧲 Dense Retrieval (Vector)

Embed query + docs into vectors. Find nearest neighbours by cosine similarity.

Great for: semantic meaning, paraphrases, multilingual

🔤 Sparse Retrieval (BM25)

Score docs by keyword frequency and rarity. No embeddings, no vectors.

Great for: exact terms, product names, IDs, code

🔀 Hybrid

Combine BM25 keyword scores with vector similarity scores for the best of both worlds.

Great for: large corpora needing both precision and recall

💡 Sahil covered Dense (Vector) retrieval. Now let's look at where it struggles — and what to use instead.

19

Where Vector Search Fails

Embeddings are powerful — but they have real blind spots.

❌ Exact terms get lost

User searches "error code E-4021". The embedding model generalises it — returns docs about "error handling" instead of the specific code.

Query: "E-4021 payment failure" Vector result: "handling payment errors" ← wrong BM25 result: "E-4021: card declined" ← correct ✓

❌ Expensive to set up

Every doc must be embedded and stored. Adding new docs means re-embedding. Needs a running embedding model 24/7.

// For 10,000 docs: 10,000 × embed() calls at ingest + pgvector index to maintain + embedding model always running

⚠️ Overkill for small corpora

200 FAQs? 50 wiki pages? Full vector infrastructure is more complexity than the problem warrants.

// Reality check 200 FAQs × 768 dims = 1.2M floats BM25 on same data = a few KB index Accuracy difference? Negligible.

These aren't edge cases — they're common real-world scenarios. This is where Vectorless shines.

20

So what is Vectorless?

Same RAG pipeline. Different retrieval step — no embeddings, no vector index.

Vector RAG: Query → embed() → cosine search on vectors → top-k → LLM

Vectorless: Query → BM25 / keyword search → top-k → LLM

✅ What stays the same

  • You still chunk your documents
  • You still retrieve top-k relevant chunks
  • You still inject them as LLM context
  • LLM still generates a grounded answer

✅ What changes

  • No embedding model needed
  • No vector index to maintain
  • Use text search you already have
  • Works with plain PostgreSQL / SQLite

💡 Vectorless = RAG with the retrieval step swapped out. Everything else is identical.

21

BM25 — The Engine Behind Vectorless

Best Match 25 — a smarter keyword ranking algorithm. Built into Elasticsearch, PostgreSQL, SQLite.

How it scores a document

Query: "cricket batting rules" Doc A: "batting rules state that..." → "batting" appears 3×, "rules" 2× → Score: 8.4 ✓ HIGH Doc B: "the umpire signals a wide" → "batting" 0×, "rules" 0× → Score: 0.1 ✗ LOW Doc C: "rules for bowling and batting" → both terms present → Score: 6.1 ✓ MEDIUM

Ranks by term frequency + how rare the term is across all docs (IDF).

Real example — box cricket bot

-- PostgreSQL FTS (BM25-style) -- No pgvector extension needed! SELECT chunk_text FROM doc_chunks WHERE to_tsvector('english', chunk_text) @@ plainto_tsquery('batting rules') ORDER BY ts_rank( to_tsvector('english', chunk_text), plainto_tsquery('batting rules') ) DESC LIMIT 3;

This is plain PostgreSQL — no extension, no embedding model, no vector index.

Real Example 1: Support Bot — Error Code Lookup

User asks: "What does error E-4021 mean?" — a case where vector search fails.

❌ Vector search gets it wrong

embed("error E-4021") → vector is close to: "handling payment errors" ✗ "common API error types" ✗ "error recovery strategies"✗ The model generalises "E-4021" into a generic "error" concept. Specific code is lost.

Embeddings are bad at exact IDs, codes, names.

✅ BM25 gets it right

BM25("error E-4021") → exact term match: "E-4021: Card declined due to insufficient funds. Ask user to retry with a different card." ✓ Term "E-4021" is rare → high IDF → BM25 ranks it top instantly.

Rare, specific terms are BM25's strength.

"E-4021" → BM25 (PostgreSQL FTS) → "E-4021: Card declined..." → LLM → ✅ "Error E-4021 means your card was declined."

Real Example 2: Internal Wiki — Deploy Question

Engineer asks: "How do we deploy to staging?" — 500 wiki pages, no vector DB.

Step 1 — BM25 retrieves top matches

Keyword search on wiki pages. Fast, no embedding needed.

BM25("deploy staging") → deploy.md score: 8.2 → ci-cd.md score: 6.1 → environments.md score: 4.8

Step 2 — Feed top chunks to LLM

Top-scored pages become context. LLM answers directly from your docs.

Context: [deploy.md content] User: "How to deploy to staging?" LLM: "Run ./deploy.sh staging then verify on Grafana at grafana.internal/staging" ✓
"deploy staging" → BM25 → top 3 pages → LLM → ✅ Grounded answer from your actual wiki

💡 BM25 score is enough to surface the right page — no reranking step needed for structured content like wikis.

❌ Vector search fails on exact codes/IDs, is expensive to set up, and is overkill for small doc sets.
